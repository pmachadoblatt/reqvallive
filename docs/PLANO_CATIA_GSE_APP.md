## 3. A dica crítica do Christopher (“no arquivo”)

Já feito com Betina/Falqueto: usar o campo de **documentação / comentários** do requisito SysML v2 como **marcador** para o app.

Exemplo (textual SysML v2):

```sysml
requirement requirement1 {
  doc /* _go_to_verification */
}
```

No Magic: propriedade **Documentation** / `doc` = `_go_to_verification` (ou tags semelhantes).

**Contrato proposto:**

| Tag em `doc` | Significado para o APP |
|--------------|-------------------------|
| `_go_to_verification` | Este requisito entra no pipeline de V&V live |
| (futuro) `_sc_threshold:...` ou JSON/MD embutido | Critério estruturado se não vier noutro atributo |
| (futuro) `_mqtt_metric:batteryLevel` | Dica de telemetria |

O app **não** varre o modelo inteiro às cegas: faz **parse** do export e filtra só o que tem a tag.

## 5. Definição do produto (o que o programa *vira*)

**Nome de trabalho:** ReqValLive como **APP de verificação GSE** entre CATIA e MQTT.

**Ciclo canónico:**

1. **Export** — ficheiro SysML v2 (ou pacote) sai do CATIA.  
2. **Select** — parser encontra `requirement` com `doc` contendo `_go_to_verification`.  
3. **Interpret** — extrai texto + SC (atributos, MD no doc, ou LLM se o SC estiver só em prosa).  
4. **Validate** — gate atual → OK/NOK.  
5a. **NOK** — relatório + ficheiro/status para o engenheiro reabrir no CATIA (UPDATE mínimo = relatório + marcador; UPDATE nativo Magic = fase 2).  
5b. **OK** — **Generate GSE** (esqueleto SysML + config de subscribe: tópico, métricas, limiares).  
6. **Subscribe / Measure** — MQTT (real ou `publish_three_drones`) → veredicto live → snapshot + relatório.  
7. **Update** — devolver status de verificação ao modelo (ficheiro de status / doc tag / plugin).

Honestidade de tese (manter): **não** prometemos runtime Magic `:=` nem Matlab no MVP; o desenho do prof é a **visão**; o APP fecha o loop **Req+SC → measure → OK/NOK** com MQTT real do CONCEPTIO.

---

### Fase B — VALIDATE alinhado ao desenho · ~1 semana

**Entregáveis**

- Pipeline: export marcado → schema Vampire / SC → **mesmo gate** ACCEPT/REJECT.  
- Resposta explícita **OK/NOK** por requisito (API + UI).  
- Artefacto `validation_status.json` (para o engenheiro / futuro UPDATE no CATIA).  
- Fechar P0 restante se ainda fizer sentido: **1.3** metadados de missão, **1.4** disclaimer SysML (“esqueleto, não simulação Magic”).

**Critério de pronto:** NOK bloqueia subscribe; OK libera montagem GSE.

---

### Fase C — GENERATE GSE + SUBSCRIBE (o teu desenho) · 1–2 semanas

**Entregáveis**

- Com OK: gerar pacote GSE mínimo:
  - esqueleto SysML (já parcial),
  - `gse_config.json` (broker, topic, métricas, SC snapshot),
  - opcional: script/checklist de medição.  
- “Montar GSE” na UI = carregar config + **iniciar subscribe** (reutilizar MQTT atual).  
- Relatório de procedimento já existente fica o laudo da medição GSE.

**Critério de pronto:** fluxo ponta a ponta sem Markdown manual: `.sysml` tagueado → OK → medir drones (publisher ou lab) → relatório.

---

### Fase D — UPDATE de volta ao CATIA · médio prazo

- Engenheiro reimporta ou cola no Magic.
